# bake-ai — V2 plan (post-MVP)

Builds on the MVP in `PLAN.md`. Big pieces: the **auto-updating vault**,
**CLI-driven project authoring with git sync**, and an optional **bake-owned chat
via ACP**.

## 1. Auto-updating vault (the distill pass + hook)

**Goal:** after every conversation, the system writes what's new into the
project's `vault/` so you never start cold — no manual note-keeping.

### Why a hook, not an in-prompt instruction

Triggering the update from a "remember to update the vault" instruction in the
recipe is unreliable: the model doesn't know when the conversation is over, and
you'll often just close the terminal. The **wrapper-on-exit hook is
deterministic** (runs every time), separates concerns (chat model helps you; a
cheap distill model only extracts facts), and is independently tunable. goose has
no native session-end hook, so the wrapper _is_ the hook (fires when the `goose`
subprocess exits). Optional hybrid: also ask the chat model to flag decisions
inline — flagging improves signal, the distill pass guarantees it gets written.

### Mechanism

New command `bake distill <project>`, auto-run by `bake chat` on exit:

1. Locate the just-ended goose **session transcript** (goose persists sessions
   with history + working dir).
2. Invoke a dedicated **distill recipe** (cheap OpenRouter model) with file access:
   - Read the transcript + current `vault/INDEX.md`.
   - Extract only **durable** new facts/decisions (ignore chit-chat).
   - For each: **update the relevant existing note**, or create a new topic note.
     Deduplicate. Keep concise. Convert relative dates to absolute; link notes.
   - Update `INDEX.md`.
3. (Optional) `git commit` the vault in the workspace so every change is
   reviewable/revertible.

### Design notes

- **Reconcile, don't append** — one topic note per subject, updated in place. Never
  one file per conversation (that's noise).
- **Distillation quality is the main failure mode** → keep the distill prompt
  strict, keep the vault git-tracked so you can review/revert what the system
  wrote, iterate against real sessions.

## 2. Project authoring via the CLI + git sync

**Goal:** manage a project's building blocks — skills, prompts, docs, context —
through `bake` commands instead of hand-editing files, and keep the workspace repo
in sync, either automatically or on demand.

### Authoring commands

All operate on the project folder via the `internal/` engine, so the future TUI
reuses the same logic:

- `bake skill add|list|remove <project> <name>` — scaffold/manage a skill snippet
  (or subrecipe) under `<project>/skills/`.
- `bake prompt edit <project>` — edit the project's role/prompt and re-render the
  goose recipe instructions from it.
- `bake doc add <project> <path|url>` — bring a reference doc into
  `<project>/docs/` (copy a file, or fetch + convert a URL to markdown).
- `bake context edit <project>` — open `.goosehints` for quick edits.

Each command mutates files through the engine and re-generates whatever backend
artifacts depend on them (e.g. the goose recipe / `.goosehints`).

### Git sync (auto + manual)

The workspace can be the user's own git repo (see `PLAN.md`). bake should be able
to commit the changes it makes:

- **Manual:** `bake sync <project>` (or `bake commit`) — stage and commit the
  project's changes with a sensible message; optional `--push`.
- **Automatic:** an opt-in config flag (e.g. `auto_commit = true` in
  `config.toml`) so any authoring command — and the vault distill pass (§1) —
  commits its own change automatically. Pushing stays manual/opt-in by default.

### Design notes

- **Engine-first:** authoring + git live in `internal/` (e.g. `internal/authoring`,
  `internal/gitsync`) with no presentation, so CLI and TUI share them.
- **Safe auto-commit:** scope commits to the touched project path, use clear
  generated messages (e.g. `bake: add skill "x" to coffee`), never auto-push, and
  no-op cleanly when the workspace isn't a git repo.
- **Secrets:** reuse the MVP `.gitignore` guardrails so authoring/sync can never
  commit a stray secret.

### Open question (to answer later)

Auto-commit assumes the **workspace** (`~/bake`) is itself a git repo, separate
from the public **tool** repo. Decide before building git sync:

- Should `bake init` (or a `bake workspace init` command) `git init` the workspace
  automatically, or leave that to the user?
- Default remote/push behavior: private repo, push opt-in only?
- Keep the tool repo and workspace repo strictly separate (recommended), confirmed?

## 3. Bake-owned chat via ACP

**Goal (optional):** instead of handing the terminal to goose (the MVP behavior),
let bake **own the chat screen** and render the whole conversation in its own
Bubble Tea + Glamour UI — full control over layout, theming, animations, and
inline decision-flagging.

### Approach: ACP (recommended)

goose can run as an **ACP (Agent Client Protocol) server**, over stdio
(`goose acp`) or HTTP/WebSocket (`goose serve`, default `127.0.0.1:3284`). bake
runs it as a subprocess and acts as an **ACP client**:

- bake sends the user's message; goose streams back structured events — assistant
  text deltas, tool calls, tool results.
- bake renders all of it in its own TUI. goose still does the heavy lifting
  (OpenRouter, MCP, tools, context); bake controls the entire screen.

Prefer `goose acp` (stdio) for a single local session — simplest and no port.

### Rejected alternatives

- **Parse headless `goose run` output** — fragile; scraping goose's human-formatted
  stdout breaks on every format change. Don't.
- **Drop goose, own the agent loop** — call OpenRouter directly and reimplement
  tool-calling/MCP. Maximum control, but that's the goose-agnostic rebuild; far
  more effort. Only if bake outgrows goose.

### Synergy with the vault (§1)

If bake owns the session via ACP, it **already holds the transcript in memory**, so
the distill pass doesn't need to locate session files — it distills what it has.
"Bake owns the chat" and "auto-updating vault" reinforce each other.

### Tradeoff / call-out

Owning the chat means bake takes on streaming, tool-call rendering, interrupts, and
error handling, and must track ACP/goose changes over time. The MVP handoff stays
the robust, zero-maintenance default; this is an opt-in upgrade, not a requirement.
